## 1. Implementation
- [ ] 1.1 **Settle the spec first — CLT-080 says "MAY" and is [P1].** Decide (owner call) between promoting it to [P0] "SHALL, in every language" and leaving it optional. "Optional" here means each product forks its own pool again, which is the duplication this repo exists to end; a per-language subset is the worst outcome of the three. No code until this is decided — the answer changes what 1.4 must cover
- [ ] 1.2 De-stale SPEC-003 §8 while it is open: CLT-080's neighbours CLT-070/071 still route through "the profile registry", deleted by `phase6_agnostic-config`. The pool takes a `Config` like every other entry point — no registry, no scheme table, no product names (PRO-010/012). Rewrite CLT-080 to state the contract (fixed N, checkout/return, lazy connect, no external pool crate) rather than pointing at "the Vectorizer pattern" — a library must not cite a product (PRO-010)
- [ ] 1.3 Read the four donors before writing anything: `Vectorizer/sdks/{rust/src/rpc/pool.rs,typescript/src/rpc/pool.ts,python/rpc/pool.py,go/rpc/pool.go}`. Extract the shape that is actually shared and the parts that are language idiom. Record what is deliberately dropped and why — the header's rejection of bb8/deadpool ("heavyweight reconnect logic that the v1 SDK doesn't need") is a decision to inherit consciously, not to copy silently
- [ ] 1.4 Implement in Rust first as the reference: `client/pool.rs` behind the existing `client` feature — fixed `max_connections`, `Semaphore` permit, idle list, lazy connect on first checkout, RAII guard returning the connection on `Drop`. Takes `Config` + `ClientConfig`, constructs the same `Client` as `connect_with`; the pool adds no wire behavior of its own
- [ ] 1.5 Port to TypeScript, Python and C# from the Rust reference — same knobs, same semantics, guard per language idiom (`Drop` / `IDisposable` / context manager / `release()` in `finally`). Divergence in behavior is a bug; divergence in idiom is required
- [ ] 1.6 Hold the layering invariants (CLT-001: "pooling is a layer above"): the single-connection `Client` API does not change; `max_in_flight` (CLT-012) stays per connection and is not re-interpreted per pool; a poisoned connection (CLT-014) is dropped from the pool rather than returned, and the next checkout connects fresh — reconnect stays CLT-030's job. Out of scope and to be refused if it creeps in: health checks, background reaping, min-idle warmup
- [ ] 1.7 Prove the mechanism the task claims, or drop the claim: a test that N sequential operations through the pool perform **one** connect + **one** handshake, not N. That — not throughput — is the thing pooling buys, and it is what both Nexus (`rest.rs:14-33`, TIME_WAIT port exhaustion) and Vectorizer's Raft (`395f7b25`, handshake per RPC) actually hit
- [ ] 1.8 **Make no quantitative claim about this work** (BEN-031). No G5 cell exercises checkout — the matrix reuses connections, and `connection-storm` is a stable tie because its cost is the kernel's. The family's one pooling number ("80%", `Vectorizer/docs/patches/v0.10.0-0.10.9.md:260`) has no artifact behind it and must not be inherited or restated as Thunder's. If a number is wanted, it needs a new BEN-011-compliant cell that opens connections per operation — a separate task, and one the instrument cannot support until `phase4_hotpath-optimization` 1.1 fixes it
