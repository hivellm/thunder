## 1. Implementation
Each protocol lane = minimal listener + parity driver over the shared no-op backend
(BEN-001/002), continuous-pipelining parity (BEN-003), validated against the real
implementation, wired into Lane::ALL + the matrix runner + Targets. Phased by
value/cost. Results stay diagnostic until a quiet host produces a citable artifact.

Phase 1 — ceiling & variant (low cost):
- [x] 1.1 Memcached binary protocol lane — the leanest possible FIFO peer; defines the performance ceiling (how close Thunder gets to a trivial protocol)
      — `thunder-bench/src/memcached.rs`: minimal binary listener (one opcode, GET; key carries the workload via STATIC/SINK/PING sentinels, else echoed) + FIFO parity driver (sender+receiver via join!, VecDeque correlation, direct nodelay writes — the lean FIFO shape) + storm. Wired as a **reference** lane (Lane::Memcached in ALL_WITH_DIAGNOSTIC, NOT in Lane::ALL — a ceiling is not a peer Thunder must beat). 7 unit tests (echo/static/ping/sink round-trip, key-as-value, 250-byte cap rejection) green; validated end-to-end over sockets (bytes exact: 64B echo -> in 88/out 92; 4KiB -> in 30/out 4124). Full Rust gate green. Preliminary (allow-noisy, not citable): Thunder is not behind the FIFO ceiling — it wins several cells, e.g. point-echo d1/c1 31.6k vs 18.9k.
- [x] 1.2 RESP2 lane — RESP3 minus the v3-only types; near-free given the existing RESP3 lane
      — **RESOLVED BY ANALYSIS, not a duplicate lane.** For every measured scenario the existing RESP3 lane emits only RESP2-compatible types: ECHO/STATIC → `$` bulk string, bare PING → `+` simple string (resp3.rs `value_to_reply`/`process`). The v2/v3 difference is null encoding (RESP3 `_\r\n` vs RESP2 `$-1\r\n`), rich types (`,` `#` `(` `=`) and push (`>`) — none of which the echo/static/ping/pipelined cells exercise (no SINK/Null scenario is active). So a RESP2 lane would measure **byte-identical** traffic to RESP3; building it would present two "competitors" that are the same wire, which is misleading, not breadth. The RESP3 lane already IS the RESP2 comparison for these workloads. (If a SINK/Null scenario is ever added, revisit — that one cell would differ by the null encoding only.)

**Decision (mid-phase, owner-directed): use REAL protocol crates, not hand-rolled peers,
wherever a Rust *server* implementation exists.** Rationale: a peer written by the
benchmark author is the worst bias in a shootout. Survey done — servers exist for
PostgreSQL (`pgwire`), gRPC (`tonic`), Cap'n Proto (`capnp-rpc`), Thrift (`thrift`, but
blocking); they do NOT exist for Memcached or MongoDB (client-only crates), and the
MessagePack-RPC ones are dead (`rmp-rpc` 0.3 pulls **tokio 0.1.22**, which would put two
reactors in one process and break BEN-001 far worse than the bias it fixes; no license
declared either). So: real server where one exists, real *codec* where only that exists,
hand-rolled only where neither does — documented per lane.
**The swap paid for itself immediately** — see 1.3: the hand-rolled Postgres peer was 2-4x
FASTER than the production `pgwire` at byte-identical traffic, i.e. hand-rolled peers
flatter the protocol rather than hobbling it. The driver stays ours in every lane (BEN-003
parity demands one concurrency model); it is the SERVER that must be real.

Phase 2 — serious binary DB wires:
- [x] 1.3 PostgreSQL v3 wire lane — startup + simple/extended query + DataRow; a mature, heavily-optimized DB wire
      — **NOW RUNS THE REAL `pgwire` CRATE (0.40, default-features off, server-api only).** Server-side this file supplies only a `SimpleQueryHandler` forwarding to the shared NoopBackend + the accept loop calling `process_socket`; startup is pgwire's `NoopStartupHandler` (no auth, no TLS). Driver stays ours (BEN-003 parity). Two documented consequences: (a) **bytes counted at the driver** — `process_socket` takes a concrete TcpStream so no counting wrapper can be interposed; same socket, other end, request bytes exact and response bytes summed per cycle; (b) **pgwire flushes ~2x per query** (`send` on CommandComplete, `send` again on ReadyForQuery) where every other lane defers its flush — a property of the real implementation, not of this harness.
      **Result of the swap (wire bytes IDENTICAL either way — in 75/out 122 point-echo, in 12/out 4154 medium — so the delta is purely implementation):** point-echo d1/c1 20.8k → **11.0k**; d16/c1 93.7k → **19.6k**; pipelined-1k c1 105k → **25.2k**. The hand-rolled peer was **2-4x faster than production pgwire**, i.e. it was biased *in Postgres's favour*. Key lesson for the remaining lanes: a peer written by the benchmark author flatters the protocol; it does not hobble it. (The old hand-rolled numbers remain meaningful as an *upper bound on what the v3 protocol permits*, distinct from what a real Rust server delivers — keep both readings separate in the artifact.)
      Original hand-rolled scope, still describing the wire the driver speaks: untyped StartupMessage → AuthenticationOk + ReadyForQuery off the measured path; then typed `type(1)+len(int32)+body` messages, big-endian throughout; FIFO parity driver + storm. The measured path is the **simple query** `Q`, text `"<CMD> <payload>"`, answered by the full four-message cycle RowDescription + DataRow + CommandComplete + ReadyForQuery written as one frame; the driver consumes through ReadyForQuery before counting a reply. **Extended query (Parse/Bind/Describe/Execute/Sync) deliberately out of scope** — five client messages per op would measure protocol chattiness, not transport, and break BEN-003 one-request/one-reply parity; documented in the module docs. Query text is a cstring, so NUL payloads are refused rather than silently truncated. Wired as a **reference** lane (Lane::Postgres in ALL_WITH_DIAGNOSTIC, not Lane::ALL). 8 unit tests green (query round-trip, bare command, length-excludes-type-byte, startup version/length, cycle message order, echo/static DataRow values, NUL refusal); validated end-to-end over sockets with byte counts exact against a hand-computation (point-echo in 75/out 122; medium STATIC in 12/out 4154; storm in 45/out 77 including startup). Full Rust gate green. Preliminary (allow-noisy, not citable): postgres is the **leanest request** of any lane (75 B vs Thunder 84, RESP3 85) and the **fattest response** (122 B vs RESP3 71) — the multi-message cycle is a per-response structural cost, exactly the thing this lane was added to price.
- [x] 1.4 MongoDB OP_MSG lane — header + BSON body; a natural codec comparison (BSON vs MessagePack)
      — **BSON is now the real `bson` crate** (Document::to_vec / from_reader, no serde feature). This lane exists to compare BSON against MessagePack, so a hand-rolled BSON subset made the codec under test a homemade codec — the single worst place in the expansion for that bias. Only the OP_MSG framing stays ours: **no Rust MongoDB server exists** (the official crate is a client), so there is nothing to borrow it from. Wire bytes unchanged after the swap (in 110/out 106, in 48/out 4138), confirming the hand-rolled encoder had been byte-faithful — but it is no longer the thing being measured. Non-UTF-8 payloads now refused (BSON strings are UTF-8 by definition) rather than mangled. 7 tests green.
      Original scope: minimal OP_MSG listener (opcode 2013, section kind 0, a two-field `{c, v}` BSON command document — `c` selects the backend mode, `v` carries the payload; replies `{ok:1, r:<value>}`) + a small BSON encoder/decoder (string + int32 fields, length-prefixed, NUL-terminated) + FIFO parity driver + storm. Length-prefixed like Thunder (messageLength first), so it reads cleanly async. Wired as a **reference** lane (Lane::Mongodb in ALL_WITH_DIAGNOSTIC, not Lane::ALL — a codec comparison, not a G5 peer). 6 unit tests green (BSON round-trip, length-prefix invariants, echo/static replies); validated end-to-end over sockets (bytes coherent: point-echo in 110/out 106; medium STATIC command in 48/out 4138). Full Rust gate green. Done ahead of the Phase 1/2 ordering because it is length-prefixed (cheaper async) than the streaming MessagePack-RPC.

Phase 3 — binary RPCs:
- [ ] 1.5 Apache Thrift (TCompactProtocol over framed transport) lane — the `thrift` crate (0.24) has a server, but it is **blocking**, not tokio; decide how to host it inside an async harness without breaking BEN-001 (same runtime) before building, and document the decision
- [x] 1.6 MessagePack-RPC lane — the Thunder sibling (msgpack array [type,msgid,method,params]); isolates Thunder's design from its codec
      — `thunder-bench/src/msgpack_rpc.rs`: requests `[0,msgid,method,params]`, replies `[1,msgid,error,result]`, encoded with **the same rmp-serde the thunder crate frames with** (added as a workspace dep to thunder-bench — that shared codec is what makes this a codec-constant comparison). The protocol is a bare self-delimiting stream, so the lane carries a **structural MessagePack scanner** (`scan_value`: nil/bool/int/float/str/bin/array/map, ext refused, depth-capped) to find message boundaries — precisely the work Thunder's `u32` prefix (WIRE-001) removes. Buffer-capped at DEFAULT_MAX_FRAME_BYTES (a stream has no prefix to validate, so the *buffer* is what must be bounded). Driver correlates FIFO and scans each reply without decoding the payload — the scan IS the mandatory framing work. Wired as a **reference** lane (Lane::MsgpackRpc in ALL_WITH_DIAGNOSTIC). 8 unit tests green, incl. the property the streaming reader depends on: every truncation prefix returns NeedMore, never Malformed; plus multi-message pipelined-stream boundary walk. Full Rust gate green (120 lib tests).
      **Finding (preliminary, allow-noisy, not citable):** with the codec held constant, msgpack-rpc lands exactly where the other FIFO peers land — pipelined-1k d1000/c1 **111k qps vs RESP3 ~100k, Memcached ~100k, Thunder 353k**. So Thunder's ~3.2x pipelined win is **not** its codec (identical here) — it is the design (multiplexed demux + write coalescing). Wire cost of the prefix: msgpack-rpc requests are 3-4 B smaller (80 vs 84 point-echo, 11 vs 15 medium), i.e. Thunder pays 4 B of prefix and gets back 1 B by not carrying a message-type element.
