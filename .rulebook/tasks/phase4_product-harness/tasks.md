## 1. Implementation
- [x] 1.1 Harness template with the three mandated scenarios — bulk ingest, small high-QPS call, pipelined polling — parameterized so any family product can point it at its own engine (BEN-040)
      — rust/thunder-bench/src/product_harness.rs. Three scenarios as data (ProductScenario: BulkIngest 1000×256B batch; PointReadHighQps tiny GET at 4 conns; PipelinedPolling tiny POLL at depth 16), each with its request shape, concurrency, and ops scaling. Parameterized over any `ProductHandler` — a product points it at its engine by implementing that one trait.
- [x] 1.2 Same-handler discipline: RPC and HTTP paths invoke the identical product handler, so the engine cancels out and the transport is the only variable at product level (BEN-040)
      — the `ProductHandler::respond` trait is the single seam. `DispatchAdapter` wraps it as the RPC lane's `thunder::server::Dispatch`; a new Dispatch-generic HTTP/1.1 listener (the shootout's is hardwired to NoopBackend and calls the sync `respond`) calls the SAME `respond`. One implementation, so the two lanes cannot diverge — the smoke test drives both and asserts the demo engine's ingest counter moved over each.
- [x] 1.3 Seed acceptance floors from Nexus's table (point read 320µs baseline -> ≤120µs target; bulk 780ms -> ≤220ms); mark them explicitly as seeds, never as results (BEN-040/BEN-031)
      — POINT_READ_SEED_* / BULK_SEED_* consts, with a compile-time assert that each target sits below its baseline. Every FloorReport carries the SEED_CAVEAT ("provisional, not a product-calibrated result … no number may be cited while the shootout substrate is unsettled, BEN-031"); the markdown stamps SEED on every floor row and the CLI prints the caveat. PipelinedPolling deliberately has NO seed (Calibrate) — the product sets it from its first run.
- [~] 1.4 Run the harness on at least one product's real engine and commit the artifact (bench-out/-equivalent: JSON + md summary, env header) — the G4 requirement
      — TEMPLATE SIDE DONE, real-engine side is product-repo work. `--product-harness` runs the whole thing against the DemoEngine and writes JSON + md with an env header; verified end to end (three scenarios, both lanes, artifact renders). Running it against a REAL engine and committing that artifact happens in a product repo (Nexus/Vectorizer/Synap), which is owner-manual and not in this tree. Also gated in spirit by BEN-031: the shootout measurement substrate is not yet trustworthy (phase4_hotpath is host-blocked), so a committed product number would be premature — the demo artifact is explicitly provisional, not the G4 artifact.
- [ ] 1.5 Products calibrate their own floors from the first measured run; calibrated floors recorded alongside the artifact (BEN-040)
      — product-repo work. The template supports it (FloorSeed::Calibrate models an un-seeded floor; every seeded floor is marked provisional so a product can override), but the calibration itself is per-product on the real engine.
- [ ] 1.6 Roll the harness out to the remaining family products (Nexus, Vectorizer, Synap each on its own engine), one committed artifact per product
      — product-repo work (separate repos, owner-manual per the product-swap policy). The template they wire is now ready.
- [ ] 1.7 Claims discipline: no quantitative product claim without its committed artifact; numbers cite artifact path/commit; end-to-end never presented as transport-isolated (BEN-031)
      — enforced in the template output (every number stamped provisional/SEED, the caveat naming BEN-031), but the discipline itself is honored at the point a product publishes a claim — product-repo work.
