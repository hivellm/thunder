## The pattern points at one thing, and it is not the wire

Thunder wins **enormously** with no concurrency (+117% / +110% at depth 1, 1 connection) and
collapses as depth and connections rise (−50% at depth 1000). The wire is not the problem — it is
measurably the most efficient lane on the wire in every single cell:

| Lane | B/op out (64 B echo) |
|---|---|
| thunder | **83** |
| resp3 | 71 (no per-request framing; its `+PONG` reply is smaller by construction) |
| bolt | 109 |
| http | **4,192** (medium-4KiB) / 158 |

HTTP moves ~2× the bytes of Thunder on point-echo and still out-throughputs it at depth 16 — so the
bottleneck is **per-request execution cost**, not bytes.

## Two suspects, both in the server hot path, both testable

1. **Spawn-per-request.** `thunder::server` acquires a semaphore permit and `tokio::spawn`s a task
   per request (SRV-003). At depth 1000 that is 1000 task spawns in flight per connection; the peer
   listeners answer inline in the read loop. This is the prime suspect for the pipelined-1k collapse.
2. **The client's serialized write mutex** (CLT-011). `driver.rs` already documents that the Thunder
   stamp at burst depth includes in-client write-lock queueing. That was written off as a latency
   artifact — but qps is down too, so it is not only the stamp.

## Fairness caveats — record before optimizing blind

- **The peer listeners are barer than Thunder's.** Thunder's does PUSH_ID checks, first-frame
  handshake gating, session state, a semaphore, spawn-per-request and per-op metrics; the RESP3/Bolt/
  HTTP peers are minimal read loops with none of it. That is a confound **against** Thunder, and it
  is not what BEN-001 asked for ("isolate the transport"). Real Redis and real Neo4j carry their own
  per-command overhead, which these peers do not. **Before treating any of these losses as a
  transport verdict, quantify how much is Thunder's server features vs its transport.**
- **The RESP3 lane is uncalibrated for throughput** (BEN-003 — see `resp3.rs`; the Docker NAT
  dominated the reference run). That weakens RESP3's numbers in general — but *not* these losses:
  BEN-003 exists to catch a peer driver that **understates** its peer and thereby flatters Thunder.
  Here RESP3 is *beating* Thunder, which is the opposite failure mode. If anything, a calibrated
  RESP3 could be faster still.
- **`pipelined-1k` Thunder p50 (14.8 ms) is not comparable** to the peers' (0.6–0.8 ms) — the stamp
  point differs (driver.rs honesty note). **qps is the comparable number in that row**, and Thunder
  loses on qps.

## Consequence (BEN-021)

Per SPEC-007 a genuine loss is **release-blocking** and spawns an optimization task, applying the
playbook in order: write coalescing / vectored writes → frame batch flush at pipeline depth →
allocation reuse in the codec → semaphore/in-flight tuning. Given the evidence above, the order
should be revisited: **spawn-per-request is not on that list and is the leading suspect.**

No quantitative performance claim may ship while this gate is red (BEN-031). The committed family
evidence (Synap ~3× RESP3 at depth 1) is *consistent* with Thunder's depth-1 dominance here — and
says nothing about the pipelined cells, which is precisely why this gate exists.
